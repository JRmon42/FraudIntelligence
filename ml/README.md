# Heimdall ‚Äî ML

This folder contains all training, scoring and Azure ML deployment assets
for the Heimdall platform's two production models:

| Model | Purpose | Runtime | Latency |
|---|---|---|---|
| `fraud-intel-ensemble` | Real-time per-transaction risk scoring | ONNX (CPU) | **p50 0.20 ms / p95 0.51 ms / p99 1.53 ms** measured locally on `Standard_F4s_v2`-class CPU (single-row request, 200-call benchmark) |
| `fraud-intel-gnn` | Nightly fraud-ring detection on heterogeneous graph | PyTorch Geometric (GPU) | Batch ‚Äî minutes per epoch |

> Latencies are well below the 5 ms inference budget. Reproduce with
> `python -m ml.score --benchmark --n 200`.

## Model approach

### Ensemble (online)
1. **Feature engineering** ‚Äî rolling 24-h per-card aggregates (count, amount
   sum, distinct-merchant count), MCC one-hot, country features, log-amount,
   hour-of-day, weekend.
2. **Base learners** ‚Äî XGBoost (`hist`), LightGBM, Logistic Regression. A
   small grid (2 configurations each) is searched on the training fold.
3. **Stacking** ‚Äî out-of-fold predictions are stacked through a meta logistic
   regression.
4. **Calibration** ‚Äî isotonic regression on the meta output keeps PSD2
   policy thresholds well-calibrated (Brier ‚âà 0.002 on synthetic).
5. **Export** ‚Äî for sub-5 ms latency the export pipeline serialises the
   preprocessor + calibrated XGB head as a single ONNX graph (opset 15 +
   `ai.onnx.ml` v3). The two non-XGB heads remain in the offline scoring
   path used for batch back-test only.

### GNN (batch)
* Heterogeneous nodes: `card`, `merchant`, `device`, `ip`
* Edges: `card-[txn]->merchant`, `card-[uses]->device`, `card-[from]->ip`
* Two-layer `HeteroConv(SAGEConv)` with self-supervised link prediction +
  supervised ring-membership head (BCE with `pos_weight`)
* Outputs per-card 16-d embeddings written to `embeddings_card.parquet` (‚Üí
  feature store) and a `ring_score` parquet (‚Üí analyst UI).
* CPU-only environments fall back to a deterministic SVD on the bipartite
  card-vs-(merchant‚à™device‚à™ip) adjacency so the smoke test runs anywhere.

## Fairness audit

`train_ensemble.py` emits a per-Nordic-country TPR table at threshold 0.5.
The platform target is **max TPR disparity ‚â§ 5 %** across the five Nordic
markets (SE, NO, DK, FI, EE), evaluated only on cohorts with **‚â• 50
fraud events** to avoid noise.

| Country | Sample size guidance | Action on >5% disparity |
|---|---|---|
| SE | ‚â• 50 fraud events | retrain with country-stratified weights |
| NO / DK / FI | ‚â• 50 fraud events | re-validate threshold per country |
| EE | smaller cohort ‚Äî monitor monthly | annual fairness review |

The smoke run on synthetic data publishes a `metrics.json` and is
inspected by the model card (`artifacts/ensemble_model_card.md`). The CI
quality bar uses ‚â• 5 events per cohort to avoid false alarms during
short fast-feedback runs.

## Monitoring

* **Data drift** ‚Äî Evidently `DataDriftPreset` over the top 20 ensemble
  features, scheduled hourly via AML pipeline. Alerts (Azure Monitor ‚Üí
  PagerDuty) on Wasserstein > 0.2 for any numeric feature or PSI > 0.25
  for any categorical.
* **Performance drift** ‚Äî rolling 7-day PR-AUC computed from labelled
  chargebacks 30 days after the scoring date. Alert on ‚â• 10 % drop vs the
  registered champion.
* **Endpoint health** ‚Äî AML built-in latency / failure-rate metrics with
  SLO p99 < 18 ms (end-to-end via the FastAPI scorer).

## EU AI Act governance (high-risk-equivalent, voluntary)

Fraud detection is excluded from *mandatory* high-risk classification by the Annex III ¬ß5(b) financial-fraud carve-out; both models are nonetheless governed to **high-risk-equivalent** standards voluntarily, producing the same artefacts:

* Model cards: `artifacts/ensemble_model_card.md`,
  `artifacts/gnn_model_card.md`
* DPIA: `docs/compliance/dpia.md`
* Lineage / catalogue: Microsoft Purview
* Human oversight: analyst review UI; manual override logged to Cosmos

## Running locally

```bash
# 1. Install conda env
conda env create -f ml/conda.yaml && conda activate fraud-intel-ml

# 2. Generate synthetic data + train ensemble (‚âà 30 s on a laptop)
python -m ml.data.synthetic_data --n 50000
python -m ml.train_ensemble --output ml/artifacts/

# 3. Train GNN (uses PyG if available, else SVD fallback for smoke)
python -m ml.train_gnn --output ml/artifacts/ --epochs 25

# 4. Benchmark online scorer
python -m ml.score --benchmark --n 500

# 5. Run unit + smoke tests
pytest ml/tests/ -v --timeout=180
```

## AML deployment commands

```bash
# 0. Login
az login && az account set -s <subscription-id>
az configure --defaults workspace=fraud-intel-ws group=fraud-intel-rg

# 1. Submit training jobs
az ml job create --file ml/aml_jobs/train_ensemble.yml
az ml job create --file ml/aml_jobs/train_gnn.yml

# 2. Register the resulting models
az ml model create --file ml/aml_jobs/register_models.yml

# 3. Create / update the managed online endpoint + blue/green deployments
az ml online-endpoint create --file ml/aml_jobs/deploy_endpoint.yml
az ml online-deployment create --file ml/aml_jobs/deploy_endpoint.yml --all-traffic

# 4. Smoke-test the endpoint
az ml online-endpoint invoke -n fraud-intel-scoring \
    --request-file ml/tests/sample_request.json
```

## Responsible AI dashboard + scorecard

`ml/train_ensemble.py` also logs the **served** sklearn pipeline (the same
calibrated XGBoost that is exported to ONNX) as an MLflow model and writes
`train_data.parquet` / `test_data.parquet`. The Responsible AI pipeline turns
those into a dashboard + downloadable PDF scorecard inside the workspace:

```bash
# One command: register the MLflow model + MLTable data assets and submit
# the RAI dashboard/scorecard pipeline (runs on the cpu-cluster).
RESOURCE_GROUP=rg-heimdall-prod-swc WORKSPACE=mlw-heimdall-prod-swc \
  ./scripts/run_rai_scorecard.sh
```

What it produces, visible under **Azure ML Studio ‚Üí Models ‚Üí
`fraud-intel-ensemble-sklearn` ‚Üí Responsible AI**:

| Insight | Source component |
|---|---|
| Feature importance (global + local) | `rai_tabular_explanation` |
| Error analysis (worst cohorts) | `rai_tabular_erroranalysis` |
| Fairness across `card_country` / `channel` | scorecard config `Fairness` block |
| Downloadable **PDF scorecard** | `rai_tabular_score_card` |

Because the workspace is private (managed VNet, `publicNetworkAccess=Disabled`),
run the script from a host with line-of-sight to it (Azure Cloud Shell, a jump
box, or a VNet-joined runner). The `cpu-cluster` it targets is defined in
`infra/modules/aml.bicep` and scales to zero when idle.

> **Honesty note:** explanations, error analysis and fairness are computed on
> the synthetic back-test set, so they are *methodologically* real but
> *substantively* illustrative until re-run on production data during the pilot.
